## [​](https://fal.ai/docs/documentation\#deploy-your-own-models)  Deploy Your Own Models

For teams that need to run custom models, proprietary pipelines, or fine-tuned variants, [fal Serverless](https://fal.ai/docs/documentation/serverless) lets you deploy on the same engine that powers the Marketplace. fal has been running this infrastructure for over 3 years, and every model on the platform goes through the same lifecycle below.

1

[Navigate to header](https://fal.ai/docs/documentation#)

Develop

A [fal.App](https://fal.ai/docs/documentation/development/app-setup) is a Python class where your `setup()` method runs once per [runner](https://fal.ai/docs/documentation/deployment/runners) to load model weights and initialize resources. Your [`@fal.endpoint`](https://fal.ai/docs/documentation/development/handle-inputs-and-outputs) methods then serve incoming [requests](https://fal.ai/docs/documentation/deployment/requests) using the initialized state. You declare hardware needs and [environment](https://fal.ai/docs/documentation/development/container-setup) alongside your code, so infrastructure is versioned with your app.

Report incorrect code

Copy

Ask AI

```
import fal

class MyModel(fal.App):
    machine_type = "GPU-H100"

    def setup(self):
        self.model = load_my_model()

    @fal.endpoint("/")
    def generate(self, prompt: str):
        return self.model(prompt)
```

2

[Navigate to header](https://fal.ai/docs/documentation#)

Test

[`fal run`](https://fal.ai/docs/documentation/development/getting-started/quick-start) spins up a cloud GPU [runner](https://fal.ai/docs/documentation/deployment/runners) and gives you a temporary URL so you can test on the same hardware you’ll use in production. It also generates a [playground UI](https://fal.ai/docs/documentation/model-apis/playground) automatically. For CI, [`AppClient`](https://fal.ai/docs/documentation/development/test-models-and-endpoints) lets you run tests against ephemeral deployments.

Deploy

[`fal deploy`](https://fal.ai/docs/documentation/deployment/deploy-to-production) creates a persistent, authenticated endpoint with autoscaling and built-in [retries](https://fal.ai/docs/documentation/serverless/reliability/retries). Every deploy creates a new revision for instant [rollbacks](https://fal.ai/docs/documentation/deployment/rollbacks). For staging and production separation, fal supports [multiple environments](https://fal.ai/docs/documentation/deployment/manage-environments) per app.

[Navigate to header](https://fal.ai/docs/documentation#)

Observe

The [dashboard](https://fal.ai/dashboard) gives you real-time logs, request-level [analytics](https://fal.ai/docs/documentation/serverless/observability/app-analytics), and [error tracking](https://fal.ai/docs/documentation/serverless/observability/error-analytics) out of the box. Trace individual requests, spot latency regressions, and monitor runner utilization. For external stacks, fal supports [Prometheus metrics](https://fal.ai/docs/documentation/serverless/observability/exporting-metrics) and [log drains](https://fal.ai/docs/documentation/serverless/observability/log-drains) to Datadog, Splunk, and Elasticsearch.

5

[Navigate to header](https://fal.ai/docs/documentation#)

Scale

fal scales [runners](https://fal.ai/docs/documentation/deployment/runners) from zero to thousands of GPUs based on demand, with a multi-layer [caching](https://fal.ai/docs/documentation/deployment/caching) system that reduces cold starts over time. [Scaling parameters](https://fal.ai/docs/documentation/deployment/scale-your-application) let you control the tradeoff: `min_concurrency` keeps runners warm, `max_concurrency` caps spend, and `concurrency_buffer` pre-warms ahead of spikes. See [optimizing cold starts](https://fal.ai/docs/documentation/serverless/optimizations/optimize-cold-starts) and [machine types](https://fal.ai/docs/documentation/deployment/machine-types) for latency-sensitive workloads.

To list on the [Marketplace](https://fal.ai/models) for broader distribution and revenue, see [publishing to the marketplace](https://fal.ai/docs/documentation/serverless/publishing-to-marketplace).

## [​](https://fal.ai/docs/documentation\#train-your-own-models)  Train Your Own Models

For training runs, fine-tuning, and workloads that need sustained GPU access, [fal Compute](https://fal.ai/docs/documentation/compute) gives you dedicated instances with full SSH control. No cold starts, no autoscaling, just raw GPU power billed at a fixed hourly rate.

## H100 SXM

Single-GPU instances for development, fine-tuning, and single-GPU training

## 8x H100 SXM

Multi-GPU instances connected over InfiniBand for distributed training

|  | Compute | Serverless |
| --- | --- | --- |
| **Best for** | Training, fine-tuning, batch jobs | API endpoints, on-demand inference |
| **Billing** | Per-hour, fixed rate | Per-second of execution |
| **Scaling** | Manual | Automatic |
| **Access** | Full SSH | Managed runners |
